### 我们的基本原则

- **开放任务，不开放控制权**
- **开放协作，不放弃风控**
- **鼓励贡献，不鼓励黑箱**
- **允许 AI 参与，不让 AI 最终裁决**

任何涉及资金权限、实盘执行、关键参数、核心账户和敏感系统的部分，都必须经过严格审核、权限隔离和人工复核。

## 我们相信什么

<div align="center">

| 可信执行 | 风险隔离 | 可验证贡献 | 公平分配 |
|---|---|---|---|
| 系统按规则运行，而不是凭个人意志修改 | 重要权限必须隔离，异常情况可人工接管 | 贡献需要证据链、版本、hash、日志与复算依据 | 分配规则应尽量透明、可审计、可申诉 |

</div>

SingularityX 避免价值分配机制演化为以下不公平状态：

- 声音更大者获得更多；
- 职位更高者天然占优；
- 更靠近决策者的成员更容易被看见；
- “显眼工作”被过度奖励；
- 底层、复用、风控、文档、审计等工作被低估。

**SingularityX 正在构建的，是一套面向长期建设的贡献证明与价值分配机制。**

## 公开挑战与赏金评审

公开挑战用于发布 SingularityX 面向可复现金融研究系统的工程与研究问题。每个挑战以题面页面、难度等级、人民币赏金上限、等级关闭规则、可复现要求和验收要求作为评审依据。对于具有竞争性的悬赏挑战，默认采用私密邮箱提交；公开 PR 仅作为贡献者自愿公开代码和报告时的可选方式：

```text
Read problem -> Implement artifact -> Prepare evidence package -> 默认私密邮箱提交 / 可选公开 PR -> Reproducibility review -> Bounty review result
```

有效提交应当可复现。GitHub 可直接打开的题面页用于展示公开背景和核心问题；详细评审与赏金认定通过挑战评审流程处理。

公开挑战的交付结果通过评审后，可按照挑战规则进入赏金认定流程。评审并不只关注单次结果，也会综合可复现性、问题完成度、边际价值、风险影响、复用性、稳定性、审计证据与提交质量。挑战成果可以形成贡献记录，并可在申请加入组织时作为能力证明和评估依据之一。

## 挑战代码提交方式

> **答复时间：** 提交或上传后五个工作日内，SingularityX 会给出答复。答复可能包括：确认进入评审、要求补充非实质性材料、初筛未通过、继续复现评测、通过某档认证或未通过等级认证。

对于具有竞争性的悬赏挑战，SingularityX 默认采用 **私密提交**。公开 Pull Request 仅适用于贡献者主动选择公开代码、报告和实现细节的情况。

### 默认方式：私密邮箱提交

除非具体挑战题面指定其他私密提交渠道，贡献者应将交付物发送至 SingularityX 官方联系邮箱：

```text
join@singularityx.tech
```

邮件标题建议：

```text
[Challenge Submission] <Challenge ID> - <Contributor Name or GitHub Username>
```

一次有效的私密提交建议包含：

1. Challenge ID；
2. 贡献者姓名和 GitHub 用户名；
3. 联系邮箱；
4. 代码压缩包、私有仓库链接或其他经认可的交付格式；
5. 可用时提供 commit hash 或代码包 SHA256 hash；
6. README 与复现说明；
7. 测试结果、实验日志、评估报告或生成输出；
8. 依赖与运行环境信息；
9. 已知限制；
10. 贡献者确认其拥有或有权提交相关成果的声明；
11. 贡献者确认提交内容不包含凭证、私有行情数据、私密账户记录、客户信息、私有交易日志、私有仓位、私有风控规则或非公开策略材料的声明。

不建议仅将源码粘贴在邮件正文中。更推荐使用压缩包、私有仓库链接或结构化 submission package。

SingularityX 将记录提交时间、提交版本、hash 或 commit 引用、评审过程、复现结果和赏金评审结果。

每位提交者对同一挑战仅有 1 次首次提交机会。仅当首次提交获得 A 档认证时，提交者可自 A 档认证确认邮件发出之时起 5 个自然日（120 小时）内，再提交 1 个完整升级版本；期限以主办方邮件系统记录的发送时间为准，逾期未提交视为放弃升级机会。每人每题合计最多 2 次提交。

升级提交仅用于申请 S 档认证，并作为新的完整、独立且不可修改的版本单独计次。升级提交达到 S 档标准的，最终按 S 档认证和结算；如 A 档奖金已经支付，仅补发 S 档奖金与已支付 A 档奖金之间的差额。同一参与者对同一挑战不重复领取 A、S 两档奖金。升级提交未达到 S 档的，原 A 档认证继续有效，不产生新的 A 档认定或奖金。

每次提交均以完整材料送达指定邮箱或指定上传渠道的时间戳为准进入评审队列。某档位认证成功并公告关闭后，不再接受该档位新的认定申请；A→S 升级机会不重新开放已经关闭的 S 档，仅可在升级提交送达时 S 档仍开放的情况下使用。关闭公告前已经收到的提交，SingularityX 仍会给出答复，但不保证该档位仍有奖金名额或继续进入该档奖金评审。

等级认证成功后，SingularityX 仅会在不披露敏感信息、私有代码、账户信息、未公开策略细节和受限数据的前提下，公示认证成功者的脱敏作品摘要、认证档位、挑战编号或经审核可公开材料，公示期为一周。

### 可选方式：公开 Pull Request

贡献者如希望主动公开提交内容，可以 Fork 仓库并提交 Pull Request。公开 PR 与私密提交采用同一技术评审标准。对于竞争性悬赏挑战，建议优先使用私密邮箱提交，以减少实现细节过早公开和重复提交争议。

通过评审后，交付成果可按照对应挑战规则进入赏金认定流程；如贡献者申请加入组织，相关成果也可作为能力证明和贡献证据之一。

## 加入组织后的价值评估与公平分配

本节适用于通过审核并正式加入 SingularityX 的成员。公开挑战赏金认定遵循挑战题面与赏金评审规则；加入组织后的价值评估侧重被采用的长期贡献、可复用组织资产、验证质量、治理纪律和公平分配。

### 公开说明范围

本首页说明的是 **制度层面的机制框架、算法纪律、治理规则、申诉流程和可公开版本框架**。具体计算设置、运行实现和个案材料，通过受控评审流程、正式协议和适用合规要求管理。

| 对外说明 | 受控治理流程管理 |
|---|---|
| 评估对象、证据要求、分配流程、规则逻辑、复核纪律、申诉机制、可公开版本框架 | 计算设置、实施细节、成员个案记录、结算记录和个案材料 |

### 方法论：评价是线性的，价值是非线性的

本框架不把“评价分”和“真实价值”混成一个概念。

- **评价是线性的**：比较、排序、解释、预算守恒和制度落地，需要稳定、透明、可复算的评分规则。
- **价值是非线性的**：真实价值可能来自协同、替代、路径依赖、时点选择、复用、市场状态和组织网络效应。
- **治理必须桥接二者**：先用线性评价建立秩序，再用非线性归因逼近真实贡献，最后通过账本约束形成可复核的分配决定。

### 六层公开框架

| 层级 | 名称 | 作用 |
|---|---|---|
| L0 | Intake / Router 层 | 根据能力、历史质量、负载和权限状态路由任务、评审权和交付路径。 |
| L1 | 协议层 | 在任务开始前定义参与方式、交付物、验收标准、证据要求、分池规则和申诉规则。 |
| L2 | 贡献记录层 | 将任务、代码、研究、文档、评审、质量工作、修正和复用记录为可审计事件。 |
| L3 | 聚合与评价层 | 将离散证据压缩成可解释的评价分、任务积分、质量档位和验证信号。 |
| L4 | 权益映射层 | 将评价与归因结果映射为奖金权重、资源优先级、项目机会、评审资格或其他约定激励。 |
| L5 | 结算与审计层 | 在约定周期内，基于评价结果、被采用成果、账本事实、申诉记录和适用规则形成结算报告。 |

### 两账本结构

本框架将“运行秩序”和“经济结算”拆开：

| 账本 | 更新频率 | 用途 | 特征 |
|---|---|---|---|
| 运行积分账本 | 高频 | 路由、资源优先权、项目机会、评审资格和协作管理。 | 简单、快、可解释，不直接等于股权、Token 或固定现金。 |
| 经济结算账本 | 低频 | 额外贡献薪酬、奖金复核、长期激励评估或其他正式安排。 | 可审计，并与被采用成果、验证可信度、复用情况、治理记录和申诉结果绑定。 |

核心原则：**运行账本解决效率，结算账本解决公平。**

## Reward Score：奖励分数示意

奖励评估不只看收益或工时，而是关注 **经过验证的边际贡献、可复用资产价值、质量、风险纪律和长期组织贡献**。

### 1) 证据门禁

No Evidence, No Allocation。一项贡献进入评估前，必须先变成可识别、可追踪、可复核的贡献资产。

典型对象包括：

| 对象 | 说明 |
|---|---|
| Contribution Asset | 代码、研究、数据、文档、产品、治理、评审或社区贡献形成的资产记录。 |
| Evidence Event | 与贡献资产相关的任务、提交、测试结果、评审、修正、复用记录或验收记录。 |
| Protocol | 当前任务适用的角色要求、验收标准、证据要求、分池规则和申诉规则。 |
| Settlement Epoch | 约定结算周期，如月度、季度或项目里程碑。 |
| Reward Proof | 贡献、评估、规则版本和分配结果的可追踪证明。 |

证据纪律：

- 来源缺失、链路薄弱或复现质量不足的证据，可以被降权或标记；
- 自证据可以记录，但不能单独按独立证据满权重处理；
- 贡献份额需要满足区间约束和守恒约束；
- 重大分配决定应绑定规则版本、评估版本、复核记录和申诉状态。

### 2) 线性评价层

评价层用于把异质证据压缩成可比较、可复算、可审计的分数。它不声称自己等于真实价值。

评价维度通常包括：

| 维度 | 含义 |
|---|---|
| Difficulty | 任务难度与能力稀缺性。 |
| Impact Scope | 对策略、模块、系统、团队、流程或公共品的影响范围。 |
| Reusability | 复用潜力与长期资产价值。 |
| Quality | 交付质量、复现质量、文档质量和维护质量。 |
| Risk Discipline | 对脆弱性、复杂度、权限敏感性、合规性和稳定性的调整。 |
| Verification Confidence | 证据链、测试、评审、审计记录和复算依据的可信度。 |
| Protocol Compliance | 是否满足任务协议、角色要求和提交纪律。 |

公开口径可以概括为：

```text
reward_score
= difficulty_base
× impact_multiplier
× reuse_multiplier
× quality_multiplier
× protocol_compliance_multiplier
× verification_confidence
× risk_discipline_adjustment
```

这个公式用于运行积分与治理输入，不等同于最终现金、股权、Token 或固定经济权益。

### 3) 分池治理

为了避免只奖励最后可见提交者，本框架按协议进行分池：

| 池子 | 用途 |
|---|---|
| Participant Pool | 奖励直接交付、研究、工程、测试、文档、评审和维护。 |
| Squad Pool | 支持长期协作、复盘和持续建设。 |
| Public Goods Pool | 奖励公共模块、复用工具、规范、Runbook、审计机制、基础设施和共享知识资产。 |
| Knowledge Asset Pool | 在适用协议下，奖励研究笔记、设计文档、ADR、评审、复盘、教程等知识资产。 |

分池比例由任务协议、成员协议、组织规则和结算周期确定。任何示例比例都只用于说明，不构成长期固定承诺。

## 分配流程

典型分配周期可能包括：

| 阶段 | 内容 |
|---|---|
| 贡献登记 | 记录贡献 ID、作者、版本、hash、关联任务和提交时间。 |
| 证据校验 | 检查来源、关联关系、可复现性、验收状态和复核记录。 |
| 协议检查 | 确认任务边界、角色要求、分池规则和申诉规则。 |
| 线性评价 | 按约定公开规则逻辑计算可比较的评价分。 |
| 非线性归因 | 评估边际贡献、反事实必要性、协作效应和稳定性约束。 |
| 质量 / 复用 / 风险复核 | 根据质量、复用性、可维护性、权限敏感性、合规性和稳定性进行调整。 |
| 账本校验 | 将评价结果与被采用成果、复核记录、修正记录和申诉状态进行对齐。 |
| 预结算报告 | 生成可复核的草案结果和理由摘要。 |
| 申诉窗口 | 允许贡献者就证据遗漏、归因错误或规则适用争议提出申诉。 |
| 人工复核 | 负责人或委员会复核 AI 建议、评估报告、异常情况和申诉。 |
| 最终结算 | 根据适用规则与协议确认分配结果。 |
| Reward Proof | 生成可追踪的奖励证明与审计记录。 |

### 非线性归因逻辑

当多人、模块或知识资产共同产生结果时，本框架使用非线性价值归因处理边际贡献和协同效应。

公开可说明的逻辑：

1. **Marginal-Contribution Anchor**：用平均边际贡献形成初始公平锚点，降低职位、提交顺序或最后可见提交者带来的偏差。
2. **Counterfactual Necessity**：分析没有某项贡献时结果是否会显著变差；浅使用、可替代、弱必要的贡献不应被强行满额分配。
3. **Quality, Risk, and Compliance Adjustment**：对复现质量弱、维护性差、规则违反或复核问题未关闭的贡献进行降权或阻断。
4. **Least-Core Stability Constraint**：检查分配是否系统性低估关键协作组合，降低结算结果的不稳定性。
5. **Hybrid Settlement**：当稳定性缺口较低时保持边际贡献锚点；当缺口较高时，按规则向更稳定的分配结果靠拢。

公开表达为：

> **以边际贡献作为公平锚点，以 Least-Core 作为联盟稳定性约束，以证据链、任务协议、质量、风险纪律、反事实必要性、账本事实和人工复核共同约束最终结算。**

### 结算条件

存在以下情况时，结算可以被延期、降权或阻断：

- 证据缺失或无法复核；
- 协议要求尚未满足；
- 贡献份额不满足守恒约束；
- 申诉尚未处理完毕；
- 存在尚未关闭的重大合规、权限或治理问题；
- 审计链、规则版本或评估报告不匹配；
- 评审利益冲突未披露。

### 组织纪律

分配系统遵循三条纪律：

1. **AI 可以辅助，人类必须负责。** AI 可以总结证据、给出初步评分建议、识别不一致并生成解释草案；最终分配决定、申诉处理、冲突仲裁、规则修订、权限审批和治理责任由人类承担。
2. **反操纵是规则的一部分。** 大量低质量资产、循环引用、夸大归因、隐瞒关键假设、不可复现结果、绕过协议流程或未披露评审利益冲突，可能导致降权、延期结算、权限限制或组织复核。
3. **人工调整必须可复核。** 调整需要记录理由、检查总量守恒、控制利益冲突、保留版本记录，并设置申诉窗口。

## 加入组织后的薪酬结构：固定薪酬 + 额外贡献薪酬

通过审核并正式加入 SingularityX 后，符合条件的成员可以获得固定薪酬，并可在贡献被采用且产生可验证价值时，进入额外贡献薪酬分配流程。

### 1. 固定薪酬

用于保障正式成员在约定角色、职责、工作周期和交付范围内的基础收入。

### 2. 额外贡献薪酬

如果成员贡献被系统采用、通过评估，并产生可验证价值，则可以进入 **额外贡献薪酬分配流程**。

额外贡献薪酬对应的价值可能来自：

- 关键系统建设；
- 高价值研究成果；
- 可复用工具或基础设施；
- 显著降低风险的风控规则；
- 明确提升组织效率的流程或文档；
- 可长期复用的策略、数据、模型或 Agent Skill；
- 对组织合作、治理、招募和生态建设产生重要影响的贡献。

> **说明**
>
> 所有涉及薪酬、奖金、分润、股权、期权、Token、积分或其他经济权益的安排，均以正式协议、适用法律、税务规则、合规要求、风控规则和组织审批结果为准。
