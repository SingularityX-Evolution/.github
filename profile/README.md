### Our Basic Principles

- **Open tasks, not open control**
- **Open collaboration, without abandoning risk control**
- **Encourage contributions, not black boxes**
- **Allow AI participation, but never let AI make the final decision**

Any area involving fund permissions, live execution, key parameters, core accounts, or sensitive systems must go through strict review, permission isolation, and human verification.

## What We Believe

<div align="center">

| Trusted Execution | Risk Isolation | Verifiable Contribution | Fair Allocation |
|---|---|---|---|
| Systems should operate according to rules, not be modified by personal discretion | Critical permissions must be isolated, with human takeover available in abnormal situations | Contributions require evidence chains, versions, hashes, logs, and recalculation basis | Allocation rules should be as transparent, auditable, and appealable as possible |

</div>

We do not want value allocation in an organization to become a system where:

- whoever speaks louder receives more;
- whoever has a higher title is naturally advantaged;
- whoever is closer to decision-makers is more easily seen;
- whoever does “visible work” is over-rewarded;
- whoever builds foundational, reusable, risk-control, documentation, or audit work is undervalued.

**SingularityX is building a proof-of-contribution and value-allocation mechanism for long-term builders.**

## Value Assessment and Fair Allocation

### One-Sentence Summary

> **Make contributions recorded, value verified, and allocation fairer.**

### Contribution Flow Illustration

```text
Contribution Submit
        ↓
Contribution ID
        ↓
Code / Document / Strategy / Data / Task Hash
        ↓
Unified Evaluation Task
        ↓
Evaluation Report
        ↓
Risk-Adjusted Marginal Value
        ↓
Fair Allocation Model
        ↓
Reward Proof
        ↓
Fixed Compensation + Additional Contribution-Based Reward
```

### 1) How Contributions Are Recorded

After a contribution is submitted, we aim to turn it into an identifiable, traceable, and reusable organizational asset, rather than leaving it in chat history, verbal discussion, or one person’s memory.

Example record fields:

```text
contribution_id
contributor_id
asset_type
version
hash
timestamp
related_task
related_repository
evaluation_task_id
review_status
reuse_count
performance_delta
risk_impact
allocation_status
audit_log
```

This means:

- there is a submission trace;
- there is version information;
- there is author and timestamp information;
- there are associated tasks and evaluation records;
- there is data on subsequent reuse and impact;
- there is an audit and review basis.

### 2) What Contributions Can Be Evaluated

We evaluate more than “strategies that directly make money.”

As long as a contribution improves the organization’s long-term capabilities, it may enter the evaluation process:

| Contribution Type | Examples | Potential Value |
|---|---|---|
| Code contribution | Backtesting modules, data interfaces, evaluation scripts, agent tools | Improve efficiency, stability, and automation |
| Strategy contribution | Factor structures, entry logic, exit logic, parameter proposals | Provide new candidate structures or improve existing ones |
| Data contribution | Cleaning rules, event labels, feature engineering, quality checks | Improve data reliability and model usability |
| Research contribution | Experiment reports, factor analysis, market structure research, failure retrospectives | Reduce trial-and-error cost and build organizational knowledge |
| Risk-control contribution | Risk rules, stress testing, anomaly detection, permission boundaries | Reduce tail risk and execution risk |
| Documentation contribution | Tutorials, standards, knowledge bases, contribution guides | Improve collaboration efficiency and lower onboarding barriers |
| Product contribution | Task systems, process design, visualization, dashboards | Improve organizational coordination and manageability |
| Governance contribution | Appeal mechanisms, audit mechanisms, allocation rules, version tracking | Improve trustworthiness and long-term fairness |
| Community contribution | Recruiting, partnerships, events, content distribution | Help the organization acquire talent, resources, and connections |

### 3) Core Principles of Value Assessment

<table>
  <tr>
    <td><b>Record first, evaluate later</b><br/>Contributions that are not recorded are difficult to allocate fairly.</td>
    <td><b>Focus on marginal value</b><br/>Assess what this contribution adds relative to the original system.</td>
  </tr>
  <tr>
    <td><b>Risk must be included</b><br/>High-return but fragile contributions should not receive unconstrained rewards.</td>
    <td><b>Consider long-term reusability</b><br/>Reusable system assets are usually more important than one-off results.</td>
  </tr>
  <tr>
    <td><b>Results should be recalculable</b><br/>Contributors should be able to understand why an evaluation was made.</td>
    <td><b>Rules should be appealable</b><br/>If there are omissions, misjudgments, or attribution errors, review and appeal mechanisms should exist.</td>
  </tr>
</table>

### 4) Evaluation Dimensions

SingularityX does not use a single metric to determine value. It evaluates contributions across multiple dimensions:

| Dimension | Description | Positive Signals | Risk / Deduction Signals |
|---|---|---|---|
| Direct Value | Direct improvement to systems, research, engineering, risk control, or collaboration | Performance improvement, efficiency improvement, fewer errors, better quality | Non-reproducible results, returns dependent on randomness |
| Marginal Contribution | What was added relative to the existing system | Clear baseline improvement, solves a key bottleneck | Highly duplicative of existing contributions, unclear added value |
| Risk Adjustment | Whether benefits introduce additional risks | Reduced drawdowns, lower anomaly risk, stronger risk controls | Increased leverage, exposure, complexity, or uncontrollability |
| Reusability | Whether the contribution can be repeatedly used, migrated, or composed | Modular design, clear documentation, stable interfaces | One-off use only, dependent on personal manual operation |
| Stability | Whether it remains stable across scenarios | Stable performance across data, markets, and time windows | Overfit to specific samples, parameters, or market conditions |
| Auditability | Whether it can be tracked, recalculated, and verified | Hashes, logs, tests, reports, version records | Missing evidence chain, impossible to review |
| Scarcity | Whether it solves a difficult or scarce-capability problem | Solves a core problem, high replacement cost | Easy to replicate, low difficulty |
| Long-Term Value | Whether it contributes to long-term organizational capability | Can become an asset, rule, or infrastructure | Short-term effectiveness but high long-term maintenance cost |
| Organizational Contribution | Whether it improves collaboration, governance, and knowledge accumulation | Helps others, improves processes, strengthens standards | Creates collaboration cost, opaque information |

## Reward Score: Reward Scoring Illustration

Rewards are not based only on returns, nor only on hours worked.

We care more about: **risk-adjusted marginal value + long-term organizational contribution**.

```text
Reward Score
= Direct Value
× Marginal Delta
× Risk Adjustment
× Reusability
× Stability
× Long-Term Coefficient
× Organizational Contribution Weight
× Verification Confidence
```

| Factor | Meaning |
|---|---|
| Direct Value | The direct effect of the contribution, such as improvements in performance, efficiency, quality, or stability |
| Marginal Delta | The new value added relative to the original system, rather than repeated existing work |
| Risk Adjustment | Adjustment for risk, fragility, complexity, permissions, and compliance impact |
| Reusability | Whether the contribution can be continuously reused by other tasks, strategies, models, or members |
| Stability | Whether the contribution remains stable across environments, samples, and stress scenarios |
| Long-Term Coefficient | Whether the contribution has long-term asset value rather than being a one-off result |
| Organizational Contribution Weight | Contribution to collaboration efficiency, knowledge accumulation, governance, and organizational capability |
| Verification Confidence | Credibility of the evidence chain, tests, audit, and recalculation results |

> **Note**
>
> The formula above is an illustration of the mechanism direction and does not constitute a fixed commitment. Specific parameters, weights, periods, scope of application, and settlement methods will be determined according to task type, member agreements, organizational rules, legal compliance, and actual evaluation results.

## Allocation Process

A typical allocation cycle may include:

| Stage | Content |
|---|---|
| Contribution Registration | Record contribution ID, author, version, hash, associated task, and submission time |
| Initial Review | Determine whether the contribution is valid, duplicative, and within task boundaries |
| Unified Evaluation | Evaluate using unified tasks, unified metrics, or unified review standards |
| Value Attribution | Analyze the contribution’s impact on results, efficiency, risk, reusability, and long-term assets |
| Risk Adjustment | Adjust for drawdowns, complexity, security, permissions, compliance, and stability |
| Allocation Calculation | Calculate the Reward Score according to rules and enter the allocation model |
| Human Review | Responsible persons or committees review AI suggestions, evaluation reports, and anomalies |
| Result Notice | Disclose evaluation basis and allocation results within permitted scope |
| Appeal Handling | Contributors may appeal omissions, misjudgments, attribution errors, or rule disputes |
| Reward Proof | Generate traceable reward proofs and audit records |

### How Team Contributions Are Handled

Many results are produced by teams rather than individuals. We aim to distinguish:

- original idea contribution;
- data and experiment contribution;
- code implementation contribution;
- risk-control and testing contribution;
- documentation and retrospective contribution;
- maintenance and iteration contribution;
- engineering contribution that migrates results into production workflows.

When necessary, we may refer to **Shapley Value**, **Least Core**, marginal contribution analysis, version tracking, and human review to reduce unfairness caused by “only rewarding the final submitter” or “only rewarding the earliest proposer.”

### How Negative Contributions Are Handled

Fair allocation is not only about rewarding positive contributions. It also requires identifying negative impact. Examples include:

- submitting non-reproducible results;
- introducing serious bugs or security risks;
- bypassing risk controls or permission boundaries;
- hiding failed experiments or key assumptions;
- creating unauditable black-box workflows;
- exaggerating contribution impact or misattributing work.

Negative impact may lead to:

- lower contribution scores;
- delayed rewards;
- re-evaluation;
- permission restrictions;
- in serious cases, organizational disciplinary procedures.

## Reward Proof

We want contributors to see not only “how much you received,” but also “why it was calculated this way.”

Example Reward Proof fields:

```text
reward_proof_id
contribution_id
contributor_id
asset_hash
evaluation_task_id
evaluation_report_hash
metrics_before
metrics_after
risk_adjusted_value
score_components
allocation_rule_version
allocation_result
reviewer_signature
appeal_status
timestamp
```

The long-term goal is to let contributors verify that:

- their contribution was included in the evaluation;
- the evaluation used agreed-upon rules;
- key data, reports, and versions were not arbitrarily altered;
- allocation results can be recalculated or audited;
- disputes can be handled through an appeal mechanism.

### AI Can Participate in Evaluation, but Cannot Make the Final Decision

| AI Can Do | Humans Must Be Responsible For |
|---|---|
| Preliminary scoring | Final allocation decisions |
| Contribution-attribution suggestions | Appeal handling |
| Code-review suggestions | Conflict arbitration |
| Strategy and model analysis | Rule revision |
| Risk explanations | Permission approval |
| Document summaries | Accountability |
| Allocation suggestions | Governance supervision |

> **Principle: AI participates in evaluation; humans are ultimately responsible.**

## Fixed Compensation + Additional Contribution Value

After review and formal admission to SingularityX, eligible members may receive:

### 1. Fixed Compensation

Fixed compensation is used to provide formal members with baseline income within agreed roles, responsibilities, work cycles, and delivery scopes.

### 2. Additional Contribution Value

If your contribution is adopted by the system, passes evaluation, and produces verifiable value, it may enter the **additional contribution value allocation process**.

Additional contribution value may come from:

- critical system construction;
- high-value research results;
- reusable tools or infrastructure;
- risk-control rules that significantly reduce risk;
- processes or documentation that clearly improve organizational efficiency;
- strategies, data, models, or agent skills that can be reused over the long term;
- contributions that materially affect partnerships, governance, recruiting, and ecosystem development.

> **Note**
>
> All arrangements involving salary, bonuses, profit sharing, equity, options, tokens, points, or other economic interests are subject to formal agreements, applicable law, tax rules, compliance requirements, risk-control rules, and organizational approval results.
