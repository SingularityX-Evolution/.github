## 背景

我们正在建设一套交易研究自动化系统，希望能够从公开交易论文中提取策略思想，复现为可运行代码，并接入真实行情与回测引擎进行验证。论文类型可能包括规则型策略、监督学习策略、强化学习策略、订单簿策略、执行策略、做市策略等。

目前最困难的地方不是让大模型生成代码，而是让生成出来的策略代码稳定接入真实回测环境。论文通常使用自定义的数据结构和实验环境，例如 MBO / order book、订单 id、tick event、reward、action space、position、transaction cost、market impact 等；而真实回测引擎有固定的行情字段、撮合规则、交易限制和账户接口。

当前大模型可以根据论文生成训练代码和推理代码，但生成代码所假设的数据接口经常不一致，导致训练失败、回测字段缺失、无交易、结果不可解释，或者不得不依赖人工补丁。我们希望解决的是这类策略复现代码和真实回测系统之间的“接口契约”问题，而不是针对某一篇论文做一次性修补。

## 核心问题

如何设计一套公开、稳定、可扩展的 `Paper Strategy Runtime Contract`，使不同交易论文复现出来的策略代码都能接入统一的训练、推理、回测接口，并且在字段缺失、时间粒度不匹配、不可交易动作或训练失败时给出明确、结构化错误，而不是静默 fallback 或生成不可解释结果？

## 公开输入与输出范围

该运行契约应覆盖以下公开接口范围：

- 规则型策略、监督学习策略、强化学习策略等不同策略形态。
- 训练入口、推理入口、行情输入、账户状态、订单输出、模型产物、日志产物等接口定义。
- tick、分钟线、日线等不同时间粒度，并能在策略声明中表达所需粒度。
- 常见交易策略所需的数据字段表达，例如 OHLCV、level-1 / level-2 order book、成交、持仓、现金、订单状态等。
- 字段缺失、标的映射失败、不可交易动作、训练失败、回测失败等情况下的结构化错误输出。
- 兼容转换的显式记录机制，避免隐藏 fallback 造成不可解释结果。

## 提交物方向

提交物可包括接口规范文档、可运行 Python 原型、示例策略、测试数据或模拟行情、运行说明、错误处理说明和兼容层设计说明。示例代码应能在本地使用公开样例数据或模拟行情数据运行，不依赖任何私有系统。

## 任务周期、悬赏金额与最终接收规则

1. 悬赏周期为 **21 个自然日**，自正式发布之时起算。
2. 截止时间后，主办方统一对所有有效提交进行评测。

4. 最终原则上仅接收 **1 名中选者** 的方案。
5. 中选者不自动获得全额，实际支付金额将根据接口规范完整度、工程可运行性、示例策略覆盖度、结构化错误能力、测试覆盖度、兼容层可审计性和报告完整度分档确定。
6. 未中选方案原则上不支付奖金；主办方不得直接使用未中选者的代码、接口定义、测试样例或核心实现细节。

## 交付标准

有效提交应至少包含以下内容：

1. **接口规范文档**
   - 清晰定义输入输出 schema、生命周期、错误码、最小示例和版本管理方式；
   - 明确训练入口、推理入口、行情输入、账户状态、订单输出、模型产物、日志产物；
   - 明确 tick、分钟线、日线等时间粒度的声明和校验方式。

2. **可运行 Python 原型**
   - 能从不同论文或不同策略目录加载策略；
   - 能通过同一接口执行训练、推理和最小回测流程；
   - 能在本地公开样例数据或模拟行情数据上运行，不依赖私有系统。

3. **示例策略覆盖**
   - 至少覆盖规则型、监督学习型、强化学习型三类策略；
   - 完整交付目标为至少 18 个示例策略，其中三类各 6 个；
   - 示例应覆盖不同数据需求和不同输出动作，不能只是同一策略的参数复制。

4. **结构化错误与禁止静默 fallback**
   - 当策略请求不存在字段、时间粒度不匹配、标的映射失败、模型文件缺失、输出非法动作、训练失败或回测失败时，必须返回明确错误；
   - 如果使用兼容转换，必须记录转换依据、影响范围、是否可关闭和审计日志；
   - 禁止用隐藏 fallback 伪装成功。

5. **测试与报告**
   - 提供自动化测试，覆盖正常执行、字段缺失、时间粒度不匹配、模型文件缺失、非法订单、训练失败、回测失败等场景；
   - 输出统一运行报告，包括策略声明、数据需求、实际输入、训练产物、推理日志、回测结果和失败原因。

6. **安全与扩展设计**
   - 优秀提交应支持策略代码沙箱执行，限制危险 import、文件访问和网络访问；
   - 给出后续接入不同回测引擎或实盘交易接口的扩展方式。

## “完全解出”的明确标准

获得全额 9,600 元，必须同时满足以下条件：

1. **工程可运行**
   - 提交完整代码仓库；
   - 包含 README.md、Dockerfile 或等价环境配置、依赖文件、配置文件、测试脚本和样例数据；
   - 主办方在隔离环境中可一键复现；
   - 能生成训练产物、推理结果、最小回测结果、错误日志和统一运行报告。

2. **接口契约完整**
   - 明确策略声明、数据需求、时间粒度、字段 schema、账户状态、订单输出、模型产物、日志产物；
   - 明确生命周期：初始化、训练、保存、加载、推理、回测调用、失败退出；
   - 提供错误码或错误对象定义，且错误信息可机器读取。

3. **三类策略充分覆盖**
   - 至少 18 个示例策略；
   - 规则型、监督学习型、强化学习型各不少于 6 个；
   - 示例策略应具有不同数据字段、时间粒度或动作空间需求，能证明契约不是为单一策略硬编码。

4. **故障可解释**
   - 字段缺失、时间粒度不匹配、模型文件缺失、非法订单、训练失败、回测失败等场景必须明确失败；
   - 不得静默 fallback；
   - 兼容转换必须记录转换依据、影响范围、是否可关闭和审计日志。

5. **测试、安全与扩展完整**
   - 自动化测试覆盖正常和异常路径；
   - 支持或清晰设计策略沙箱，限制危险 import、文件访问和网络访问；
   - 给出接入不同回测引擎或实盘交易接口的扩展方式。

## 无效提交认定

以下情况可判定为无效提交或不通过：

- 无法运行、无法复现或缺少核心代码；
- 仅提交文字方案、PPT、截图或无法验证的结论；
- 没有统一训练 / 推理 / 回测入口；
- 示例策略不足，或三类策略覆盖缺失；
- 字段缺失、时间粒度不匹配或非法动作时静默修正；
- 用隐藏 fallback 伪装运行成功；
- 错误输出不可机器读取，无法定位失败原因；
- 依赖私有系统、私有行情或不可公开验证的接口；
- 存在明显危险代码执行风险且无隔离说明。

## 脱敏说明

该题目不要求接入任何特定公司系统、交易账号或私有行情数据。参赛者可以使用公开样例数据或自行生成模拟行情数据，只需证明接口设计可以稳定承接不同类型交易策略。
