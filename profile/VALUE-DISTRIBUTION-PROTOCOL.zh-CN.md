# 价值评估与分配公开框架

| 项目 | 内容 |
|---|---|
| 规则版本 | V2.0.0（第二版） |
| 发布日期 | 2026-06-12 |
| 规则解释主体 | 零界演化 |


本文件说明 SingularityX 正式成员的长期贡献如何被记录、验证、评价、归因、结算和申诉。公开挑战赏金认定遵循挑战题面与赏金评审规则；加入组织后的价值评估侧重被采用的长期贡献、可复用组织资产、验证质量、治理纪律和公平分配。

本文件说明的是制度层面的机制框架、算法纪律、治理规则、申诉流程和可公开版本框架。具体计算设置、运行实现和个案材料，通过受控评审流程、正式协议和适用合规要求管理。

## 1. 方法论

评价是线性的，价值是非线性的。

- 线性评价用于比较、排序、解释、预算守恒和制度落地，需要稳定、透明、可复算。
- 非线性价值可能来自协同、替代、路径依赖、时点选择、复用、市场状态和组织网络效应。
- 治理流程先用线性评价建立秩序，再用非线性归因逼近真实贡献，最后通过账本约束形成可复核的分配决定。

## 2. 六层公开框架

| 层级 | 名称 | 作用 |
|---|---|---|
| L0 | Intake / Router 层 | 根据能力、历史质量、负载和权限状态路由任务、评审权和交付路径。 |
| L1 | 协议层 | 在任务开始前定义参与方式、交付物、验收标准、证据要求、分池规则和申诉规则。 |
| L2 | 贡献记录层 | 将任务、代码、研究、文档、评审、质量工作、修正和复用记录为可审计事件。 |
| L3 | 聚合与评价层 | 将离散证据压缩成可解释的评价分、任务积分、质量档位和验证信号。 |
| L4 | 权益映射层 | 将评价与归因结果映射为奖金权重、资源优先级、项目机会、评审资格或其他约定激励。 |
| L5 | 结算与审计层 | 在约定周期内，基于评价结果、被采用成果、账本事实、申诉记录和适用规则形成结算报告。 |

## 3. 两账本结构

| 账本 | 更新频率 | 用途 | 特征 |
|---|---|---|---|
| 运行积分账本 | 高频 | 路由、资源优先权、项目机会、评审资格和协作管理。 | 简单、快速、可解释，不直接等于股权、Token 或固定现金。 |
| 经济结算账本 | 低频 | 额外贡献薪酬、奖金复核、长期激励评估或其他正式安排。 | 可审计，并与被采用成果、验证可信度、复用情况、治理记录和申诉结果绑定。 |

核心原则：运行账本解决效率，结算账本解决公平。

## 4. 公开算法纪律

### 4.1 证据门禁

No Evidence, No Allocation。一项贡献进入评估前，必须先变成可识别、可追踪、可复核的贡献资产。

典型对象包括：

| 对象 | 说明 |
|---|---|
| Contribution Asset | 代码、研究、数据、文档、产品、治理、评审或社区贡献形成的资产记录。 |
| Evidence Event | 与贡献资产相关的任务、提交、测试结果、评审、修正、复用记录或验收记录。 |
| Protocol | 当前任务适用的角色要求、验收标准、证据要求、分池规则和申诉规则。 |
| Settlement Epoch | 约定结算周期，如月度、季度或项目里程碑。 |
| Reward Proof | 贡献、评估、规则版本和分配结果的可追踪证明。 |

证据纪律：来源缺失、链路薄弱或复现质量不足的证据，可以被降权或标记；自证据可以记录，但不能单独按独立证据满权重处理；贡献份额需要满足区间约束和守恒约束；重大分配决定应绑定规则版本、评估版本、复核记录和申诉状态。

### 4.2 线性评价层

评价层用于把异质证据压缩成可比较、可复算、可审计的分数。它不声称自己等于真实价值。

公开口径可以概括为：

```text
reward_score
= difficulty_base
× impact_multiplier
× reuse_multiplier
× quality_multiplier
× protocol_compliance_multiplier
× verification_confidence
× risk_discipline_adjustment
```

该公式用于运行积分与治理输入，不等同于最终现金、股权、Token 或固定经济权益。

### 4.3 分池治理

为了避免只奖励最后可见提交者，本框架按协议进行分池：

| 池子 | 用途 |
|---|---|
| Participant Pool | 奖励直接交付、研究、工程、测试、文档、评审和维护。 |
| Squad Pool | 支持长期协作、复盘和持续建设。 |
| Public Goods Pool | 奖励公共模块、复用工具、规范、Runbook、审计机制、基础设施和共享知识资产。 |
| Knowledge Asset Pool | 在适用协议下，奖励研究笔记、设计文档、ADR、评审、复盘、教程等知识资产。 |

分池比例由任务协议、成员协议、组织规则和结算周期确定。任何示例比例都只用于说明，不构成长期固定承诺。

### 4.4 非线性价值归因

当多人、模块或知识资产共同产生结果时，本框架使用非线性价值归因处理边际贡献和协同效应。

公开可说明的逻辑：

1. Marginal-Contribution Anchor：用平均边际贡献形成初始公平锚点，降低职位、提交顺序或最后可见提交者带来的偏差。
2. Counterfactual Necessity：分析没有某项贡献时结果是否会显著变差；浅使用、可替代、弱必要的贡献不应被强行满额分配。
3. Quality, Risk, and Compliance Adjustment：对复现质量弱、维护性差、规则违反或复核问题未关闭的贡献进行降权或阻断。
4. Least-Core Stability Constraint：检查分配是否系统性低估关键协作组合，降低结算结果的不稳定性。
5. Hybrid Settlement：当稳定性缺口较低时保持边际贡献锚点；当缺口较高时，按规则向更稳定的分配结果靠拢。

正式公开口径是：以边际贡献作为公平锚点，以 Least-Core 作为联盟稳定性约束，以证据链、任务协议、质量、风险纪律、反事实必要性、账本事实和人工复核共同约束最终结算。

## 5. 结算与申诉流程

典型流程：

```text
贡献登记
→ 证据校验
→ 协议检查
→ 线性评价
→ 非线性归因
→ 质量 / 复用 / 风险复核
→ 账本校验
→ 预结算报告
→ 申诉窗口
→ 人工复核
→ 最终结算
→ Reward Proof
```

存在证据缺失或无法复核、协议要求尚未满足、贡献份额不满足守恒约束、申诉尚未处理完毕、重大合规/权限/治理问题尚未关闭、审计链或规则版本不匹配、评审利益冲突未披露等情况时，结算可以被延期、降权或阻断。

## 6. 组织纪律

AI 可以总结证据、给出初步评分建议、识别不一致并生成解释草案；最终分配决定、申诉处理、冲突仲裁、规则修订、权限审批和治理责任由人类承担。

反操纵是规则的一部分。大量低质量资产、循环引用、夸大归因、隐瞒关键假设、不可复现结果、绕过协议流程或未披露评审利益冲突，可能导致降权、延期结算、权限限制或组织复核。

人工调整必须可复核。调整需要记录理由、检查总量守恒、控制利益冲突、保留版本记录，并设置申诉窗口。

## 7. Reward Proof

Reward Proof 的目标是让贡献者不仅知道获得了多少，还知道为什么这样计算。

可公开字段示意：

```text
reward_proof_id
contribution_asset_id
evaluation_round_id
rule_version
allocation_policy_hash
evidence_summary_hash
evaluation_report_hash
settlement_epoch
allocation_result
review_status
appeal_status
timestamp
```

Reward Proof 不是 Token、证券、股权、支付工具或固定现金价值；它是贡献评估与分配过程的证明记录。

| 层级 | 可公开定位 | 公开说明 |
|---|---|---|
| Public Homepage | 对外 GitHub 入口 | 公开挑战、组织加入、贡献证明、价值评估框架、治理纪律和合规边界。 |
| Contribution Asset Layer | 贡献记录层 | 将代码、研究笔记、设计文档、评审、Runbook、复盘和社区工作转化为贡献资产。 |

| Governance Hardening Layer | 规则质量层 | 加入规则版本追踪、政策检查、利益冲突控制、审计记录和可申诉的 Reward Proof。 |
