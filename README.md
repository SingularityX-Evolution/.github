|---|---|---|---|

## 价值评估与公平分配

### 一句话概括

> **让贡献被记录，让价值被验证，让分配更公平。**

### 贡献流转示意

```text
Contribution Submit
        ↓
Contribution ID
        ↓
Code / Document / Strategy / Data / Task Hash
        ↓
Unified Evaluation Task
        ↓
Evaluation Report
        ↓
Risk-Adjusted Marginal Value
        ↓
Fair Allocation Model
        ↓
Reward Proof
        ↓
Fixed Compensation + Additional Contribution-Based Reward
```

### 1) 贡献如何被记录

一项贡献提交后，会尽量转化为可识别、可追踪、可复用的组织资产，而不是停留在聊天记录、口头讨论或某个人的记忆里。

典型记录字段示意：

```text
contribution_id
contributor_id
asset_type
version
hash
timestamp
related_task
related_repository
evaluation_task_id
review_status
reuse_count
performance_delta
risk_impact
allocation_status
audit_log
```

这意味着：

- 有提交痕迹；
- 有版本信息；
- 有作者与时间；
- 有关联任务与评估记录；
- 有后续复用与影响数据；
- 有审计和复核依据。

### 2) 什么样的贡献可以被评估

我们评估的，不只是“直接赚钱的策略”。

只要它能提升组织的长期能力，就可能进入评估流程：

| 贡献类型 | 示例 | 可能产生的价值 |
|---|---|---|
| 代码贡献 | 回测模块、数据接口、评估脚本、Agent 工具 | 提高效率、稳定性、自动化程度 |
| 策略贡献 | 因子结构、入场逻辑、出场逻辑、参数方案 | 带来新的候选结构或提升已有方案 |
| 数据贡献 | 清洗规则、事件标签、特征工程、质量检测 | 提升数据可靠性与模型可用性 |
| 研究贡献 | 实验报告、因子分析、市场结构研究、失败复盘 | 降低试错成本，沉淀组织认知 |
| 风控贡献 | 风险规则、压力测试、异常检测、权限边界 | 降低尾部风险与执行风险 |
| 文档贡献 | 教程、规范、知识库、贡献指南 | 提高协作效率，降低进入门槛 |
| 产品贡献 | 任务系统、流程设计、可视化、看板 | 提高组织协同与可管理性 |
| 治理贡献 | 申诉机制、审计机制、分配规则、版本追踪 | 提高可信度与长期公平性 |
| 社区贡献 | 招募、合作、活动、内容传播 | 帮助组织获取人才、资源与连接 |

### 3) 价值评估的核心原则

<table>
  <tr>
    <td><b>先记录，再评估</b><br/>没有被记录的贡献，很难被公平分配。</td>
    <td><b>看边际价值</b><br/>关注这项贡献相对原系统到底新增了什么。</td>
  </tr>
  <tr>
    <td><b>风险必须计入</b><br/>高收益但高脆弱性的贡献，不应得到无约束奖励。</td>
    <td><b>看长期复用性</b><br/>可重复使用的系统资产，通常比一次性结果更重要。</td>
  </tr>
  <tr>
    <td><b>结果要可复算</b><br/>贡献者应能知道为什么这样评估。</td>
    <td><b>规则要可申诉</b><br/>如果遗漏、误判、归因错误，应有复核和申诉机制。</td>
  </tr>
</table>

### 4) 评估维度

SingularityX 不用单一指标决定价值，而是综合多维度评估：

| 维度 | 说明 | 正向信号 | 风险 / 扣分信号 |
|---|---|---|---|
| 直接价值 | 对系统、研究、工程、风控或协作带来的直接提升 | 性能提升、效率提升、错误减少、质量改善 | 结果不可复现、收益依赖偶然性 |
| 边际贡献 | 相比原有系统新增了什么 | 明确改善 baseline，解决关键瓶颈 | 与已有贡献高度重复，新增价值不清晰 |
| 风险调整 | 收益是否伴随额外风险 | 降低回撤、降低异常风险、增强风控 | 增加杠杆、敞口、复杂度或不可控性 |
| 可复用性 | 能否被多次调用、迁移或组合 | 模块化、文档清晰、接口稳定 | 只能一次性使用，依赖个人手工操作 |
| 稳定性 | 在不同场景下是否稳定 | 跨数据、跨市场、跨时间窗口表现稳定 | 对特定样本、参数或行情过拟合 |
| 可审计性 | 是否能被追踪、复算和验证 | 有 hash、日志、测试、报告、版本记录 | 缺少证据链，无法复核 |
| 稀缺性 | 是否解决高难度或稀缺能力问题 | 解决核心难题，替代成本高 | 容易复制，难度较低 |
| 长期价值 | 是否有助于组织长期能力建设 | 可沉淀为资产、规则或基础设施 | 短期有效但长期维护成本高 |
| 组织贡献 | 是否改善协作、治理和知识沉淀 | 帮助他人、改进流程、完善规范 | 制造协作成本、信息不透明 |

## Reward Score：奖励分数示意

奖励不是只看收益，也不是只看工时。

我们更关注：**风险调整后的边际价值 + 长期组织贡献**。

```text
Reward Score
= Direct Value
× Marginal Delta
× Risk Adjustment
× Reusability
× Stability
× Long-Term Coefficient
× Organizational Contribution Weight
× Verification Confidence
```

| 因子 | 含义 |
|---|---|
| Direct Value | 贡献带来的直接效果，如性能、效率、质量或稳定性提升 |
| Marginal Delta | 相对原系统的新增价值，而不是重复已有工作 |
| Risk Adjustment | 对风险、脆弱性、复杂度、权限和合规影响进行调整 |
| Reusability | 贡献是否可以被其他任务、策略、模型或成员持续复用 |
| Stability | 贡献是否在不同环境、样本和压力场景下保持稳定 |
| Long-Term Coefficient | 贡献是否具有长期沉淀价值，而非一次性结果 |
| Organizational Contribution Weight | 对协作效率、知识沉淀、治理和组织能力的贡献 |
| Verification Confidence | 证据链、测试、审计和复算结果的可信度 |

> **说明**
>
> 上述公式是机制方向示意，不构成固定承诺。具体参数、权重、周期、适用范围与结算方式，会根据任务类型、成员协议、组织规则、法律合规和实际评估结果确定。

## 分配流程

典型分配周期可能包括：

| 阶段 | 内容 |
|---|---|
| 贡献登记 | 记录贡献 ID、作者、版本、hash、关联任务和提交时间 |
| 初步评审 | 判断贡献是否有效、是否重复、是否符合任务边界 |
| 统一评估 | 使用统一任务、统一指标或统一审查标准进行评估 |
| 价值归因 | 分析贡献对结果、效率、风险、复用性和长期资产的影响 |
| 风险调整 | 对回撤、复杂度、安全、权限、合规和稳定性进行调整 |
| 分配计算 | 根据规则计算 Reward Score，并进入分配模型 |
| 人类复核 | 负责人或委员会复核 AI 建议、评估报告和异常情况 |
| 结果通知 | 在允许范围内披露评估依据和分配结果 |
| 申诉处理 | 贡献者可对遗漏、误判、归因错误或规则争议提出申诉 |
| Reward Proof | 生成可追踪的奖励证明与审计记录 |

### 团队贡献如何处理

很多成果不是单人完成，而是多人共同产生的。我们会尽量区分：

- 原始想法贡献；
- 数据与实验贡献；
- 代码实现贡献；
- 风控与测试贡献；
- 文档与复盘贡献；
- 维护与迭代贡献；
- 将成果迁移到生产流程中的工程贡献。

在需要时，可以参考 **Shapley Value**、**Least Core**、边际贡献分析、版本追踪与人工复核等方式，减少“只看最终提交者”或“只看最早提出者”的不公平。

### 负面贡献如何处理

公平分配不只奖励正贡献，也要识别负面影响。例如：

- 提交不可复现结果；
- 引入严重 bug 或安全风险；
- 绕过风控或权限边界；
- 隐瞒失败实验或关键假设；
- 制造不可审计的黑箱流程；
- 夸大贡献影响或错误归因。

负面影响可能导致：

- 贡献分数降低；
- 奖励延期；
- 重新评估；
- 权限限制；
- 严重情况下进入组织纪律流程。

## Reward Proof：奖励证明

我们希望贡献者不仅看到“你获得了多少”，还能看到“为什么这样计算”。

Reward Proof 示例字段：

```text
reward_proof_id
contribution_id
contributor_id
asset_hash
evaluation_task_id
evaluation_report_hash
metrics_before
metrics_after
risk_adjusted_value
score_components
allocation_rule_version
allocation_result
reviewer_signature
appeal_status
timestamp
```

长期目标是让贡献者能够验证：

- 自己的贡献确实被纳入评估；
- 评估使用的是约定规则；
- 关键数据、报告和版本没有被随意篡改；
- 分配结果可以被复算或审计；
- 存在争议时，可以通过申诉机制处理。

### AI 可以参与评估，但不能最终裁决

| AI 可以做 | 人类必须负责 |
|---|---|
| 初步评分 | 最终分配决定 |
| 贡献归因建议 | 申诉处理 |
| 代码审查建议 | 冲突仲裁 |
| 策略和模型分析 | 规则修订 |
| 风险解释 | 权限审批 |
| 文档总结 | 责任承担 |
| 分配建议 | 治理监督 |

> **原则：AI 参与评估，人类最终负责。**

## 固定薪酬 + 额外贡献价值

通过审核并正式加入 SingularityX 后，符合条件的成员可以获得：

### 1. 固定薪酬

用于保障正式成员在约定角色、职责、工作周期和交付范围内的基础收入。

### 2. 额外贡献价值

如果你的贡献被系统采用、通过评估，并产生可验证价值，则可以进入 **额外贡献价值分配流程**。

额外贡献价值可能来自：

- 关键系统建设；
- 高价值研究成果；
- 可复用工具或基础设施；
- 显著降低风险的风控规则；
- 明确提升组织效率的流程或文档；
- 可长期复用的策略、数据、模型或 Agent Skill；
- 对组织合作、治理、招募和生态建设产生重要影响的贡献。

> **说明**
>
> 所有涉及薪酬、奖金、分润、股权、期权、Token、积分或其他经济权益的安排，均以正式协议、适用法律、税务规则、合规要求、风控规则和组织审批结果为准。
